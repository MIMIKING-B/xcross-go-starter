# gf init -r 私有仓库模板使用指南

## 概述

`gf init -r` 支持通过指定远程 Git 仓库作为自定义项目模板，自动完成以下工作：

1. 下载远程仓库到临时目录
2. 复制到目标目录，删除 `.git`、`go.work`、`go.work.sum`
3. 替换 `go.mod` 中的 module 名为新项目名
4. 通过 AST 分析替换所有 Go 文件中的 import 路径
5. 格式化 Go 文件
6. 执行 `go mod tidy`

但在使用私有仓库（如 GitHub 私有仓库、阿里云 Codeup 等）时，会遇到一系列配置问题。本文档记录了这些问题的根因分析及正确的使用方法。

---

## 一、问题现象与根因分析

### 问题 1：SSH 格式 URL 不支持

**现象：**

```bash
gf init -r git@github.com:MyOrg/my-template.git new-project
# 报错：malformed module path "git": missing dot in first path element
```

**根因：**

`gf init -r` 参数要求使用 **Go module 风格路径**（如 `github.com/owner/repo`），而非 SSH 格式。源码中有两条下载路径：

- **Go Module 路径**：调用 `go get`，要求路径为标准 module 格式
- **Git Clone 路径**：`ParseGitURL()` 只识别 `https://` 前缀，会自动拼成 `https://github.com/owner/repo.git`

两种路径均不支持 `git@` SSH 格式。

**正确写法：**

```bash
gf init -r github.com/MyOrg/my-template new-project
```

---

### 问题 2：go.mod module 路径与仓库地址不一致

**现象：**

```bash
# 报错：module declares its path as: my-template
#       but was required as: github.com/MyOrg/my-template
```

**根因：**

Go 的模块系统强制要求 `go.mod` 中声明的 module path 必须与请求路径一致。`go get github.com/MyOrg/my-template` 要求 `go.mod` 中写的是：

```go
module github.com/MyOrg/my-template
```

而不是：

```go
module my-template
```

**解决方案：**

修改模板仓库的 `go.mod` 首行为完整仓库路径，并同步修改所有 Go 文件中引用自身包的 import 路径：

```go
// 修改前
import "my-template/internal/xxx"

// 修改后
import "github.com/MyOrg/my-template/internal/xxx"
```

> 初始化新项目时，`gf init` 的 AST 替换逻辑会自动将 `github.com/MyOrg/my-template` 替换为新项目的 module 路径，所以模板仓库使用完整路径是完全正确的。

---

### 问题 3：Go 代理缓存与校验和验证失败（国内环境核心问题）

**现象：**

```bash
# 现象 A：拉取到旧版本（伪版本号未更新）
# 现象 B：校验和验证超时
# 报错：verifying module: ... Get "https://sum.golang.org/lookup/...": dial tcp [...]:443: i/o timeout
```

**根因：**

`go get` 默认经过两层外部服务：

```
go get → GOPROXY (proxy.golang.org) → 下载模块
       → sum.golang.org → 校验哈希值
```

在国内环境下，这两个服务均不稳定：
- **代理缓存**：`proxy.golang.org` 会缓存模块，推送更新后不会立即同步
- **校验和超时**：`sum.golang.org` 经常无法访问，导致验证超时

**为什么以下方案不可靠：**

| 方案 | 问题 |
|------|------|
| `GOPROXY=direct` | 绕过了代理，但仍会走 `sum.golang.org` 校验，超时依旧 |
| `GOPROXY=off` | 完全禁用模块下载，连版本解析都失败 |
| `GONOSUMCHECK=...` 环境变量 | 在 `gf init` 子进程中模式匹配不可靠，对含大写字母的路径（如 `MyOrg`）可能不生效 |

**正确解决方案：使用 `go env -w GOPRIVATE`**

```bash
go env -w GOPRIVATE=github.com/MyOrg
```

`GOPRIVATE` 是 Go 官方推荐的私有模块配置方式，通过写入 Go 自身配置文件（`$HOME/.config/go/env`）持久化存储，所有 `go` 子命令都会读取。它同时隐式设置：

- **GONOSUMDB** — 跳过校验和数据库查询
- **GONOSUMCHECK** — 跳过校验和验证
- **自动使用 direct 模式** — 绕过代理直接从 VCS 下载

一步到位，彻底解决代理缓存和校验和两个问题。

## 二、正确使用步骤

### 前提条件

- 已安装 Go（1.23+）
- 已安装 `gf` CLI 工具
- 模板仓库已推送到 Git 远程仓库
- 模板仓库的 `go.mod` module 路径与仓库 URL 一致

### 步骤 1：配置 GOPRIVATE（一次性，必做）

```bash
# 设置 GOPRIVATE，值为你仓库的组织路径前缀
# 支持逗号分隔多个路径
go env -w GOPRIVATE=github.com/MyOrg

# 如果同时使用阿里云 Codeup
go env -w GOPRIVATE=github.com/MyOrg,codeup.aliyun.com

# 验证设置
go env GOPRIVATE
```

> ⚠️ 这是最关键的一步。不配置 GOPRIVATE，后续所有操作都会因代理或校验和问题失败。

### 步骤 2：配置 Git 认证

`gf init` 内部通过 `git clone` 下载仓库，需要本机 Git 能访问远程仓库。

**方式 A — SSH Key（推荐，适用于已配置 SSH 密钥的场景）：**

```bash
# 让 Git 自动将 HTTPS 请求转为 SSH
git config --global url."git@github.com:".insteadOf "https://github.com/"
```

**方式 B — HTTPS Token（适用于无 SSH 密钥的场景）：**

```bash
# 使用 GitHub Personal Access Token
git config --global url."https://<user>:<token>@github.com/".insteadOf "https://github.com/"
```

**方式 C — 阿里云 Codeup 认证：**

```bash
git config --global url."https://<user>:<token>@codeup.aliyun.com/".insteadOf "https://codeup.aliyun.com/"
```

### 步骤 3：确保模板仓库 go.mod 正确

模板仓库的 `go.mod` 首行必须是完整仓库路径：

```go
module github.com/MyOrg/my-template

go 1.23.0
```

所有 Go 文件中引用自身包的 import 路径也必须使用完整路径：

```go
import "github.com/MyOrg/my-template/internal/service"
```

### 步骤 4：使用 gf init 初始化项目

```bash
# 基本用法（module 名默认为 new-project）
gf init -r github.com/MyOrg/my-template new-project

# 指定自定义 module 路径
gf init -r github.com/MyOrg/my-template new-project -g "github.com/neworg/new-project"

# 指定版本标签
gf init -r github.com/MyOrg/my-template@v0.1.0 new-project

# 交互式选择版本
gf init -r github.com/MyOrg/my-template new-project -s

# 升级依赖到最新版本
gf init -r github.com/MyOrg/my-template new-project -u
```

### 步骤 5：验证结果

```bash
cd new-project

# 确认 module 名已替换为新项目名
head -1 go.mod

# 确认旧模板的 import 路径已全部替换（应无输出）
grep -r "github.com/MyOrg/my-template" --include="*.go" .

# 确认编译通过
go build ./...
```

## 三、使用阿里云 Codeup 私有仓库

如果模板仓库托管在阿里云 Codeup，需要额外配置：

```bash
# 1. 将 Codeup 加入 GOPRIVATE
go env -w GOPRIVATE=github.com/MyOrg,codeup.aliyun.com

# 2. 配置 Codeup Git 认证
git config --global url."https://<user>:<token>@codeup.aliyun.com/".insteadOf "https://codeup.aliyun.com/"

# 3. 使用（URL 格式为 codeup.aliyun.com/<namespace>/<repo>）
gf init -r codeup.aliyun.com/<namespace>/<repo> new-project
```

## 四、常见问题速查

| 问题 | 错误信息 | 解决方案 |
|------|----------|----------|
| SSH URL 格式 | `malformed module path "git"` | 改用 `github.com/owner/repo` 格式 |
| module 路径不匹配 | `module declares its path as: xxx but was required as: yyy` | 修改模板 `go.mod` 的 module 行为完整仓库路径 |
| 代理缓存旧版本 | 版本号未更新 | `go env -w GOPRIVATE=...` |
| 校验和验证超时 | `sum.golang.org: i/o timeout` | `go env -w GOPRIVATE=...` |
| GOPROXY=off 失败 | `module lookup disabled` | 改用 `GOPRIVATE`，不要用 `GOPROXY=off` |
| GONOSUMCHECK 不生效 | 仍访问 sum.golang.org | 改用 `go env -w GOPRIVATE` |
| 目标目录非空 | `target directory xxx is not empty` | 使用空目录或当前目录 `.` |

---

## 五、配置速记

```bash
# 一次性配置（推荐）
go env -w GOPRIVATE=github.com/MyOrg,codeup.aliyun.com
git config --global url."git@github.com:".insteadOf "https://github.com/"

# 初始化项目
gf init -r github.com/MyOrg/my-template new-project
```

## 附录：gf init -r 源码执行流程

```
cInit.Index()                                    [cmd_init.go]
  └─ initFromRemote()                            [cmd_init.go]
       └─ geninit.Process()                      [geninit.go]
            ├─ IsSubdirRepo(repo)?
            │   ├─ true  → processGitSubdir()    [git clone + sparse checkout]
            │   └─ false → processGoModule()     [go get]
            │        ├─ GetLatestVersion()        [go list -m -json]
            │        ├─ downloadTemplate()        [go get repo@version]
            │        ├─ generateProject()         [复制 + 替换 go.mod + AST 替换 import]
            │        └─ tidyDependencies()        [go mod tidy]
            │
            └─ go get 失败后:
                 ├─ IsSubdirRepo()? → true  → 回退到 processGitSubdir()
                 └─ IsSubdirRepo()? → false → ❌ 直接报错，无 git clone 回退
```

> 注意：对于完整仓库 URL（3 段路径 `host/owner/repo`），`IsSubdirRepo()` 始终返回 `false`，因此 `go get` 失败后没有 `git clone` 回退机制。这是代码层面的一个可改进点。
