## 目录结构

> 本文档描述 **xcross-go-starter** 项目的完整目录架构及各目录/文件的职责说明。
> 项目基于 [GoFrame](https://goframe.org) 框架搭建，遵循其标准工程规范并做了部分扩展。

### 二、目录/文件职责说明

| 目录/文件 | 描述 |
|---------|------|
| **api** | 对外接口定义层。仅包含数据结构（Req/Res），无业务逻辑。使用 `g.Meta` 标签定义路由、方法、参数验证规则。GoFrame 会根据此目录生成 Controller 和 OpenAPI 文档。 |
| --- api | 前台通用 API 接口定义（预留，目前为空）。 |
| --- home/base | 前台页面接口定义，如首页 `SiteIndexReq`/`SiteIndexRes`。 |
| **hack** | 存放项目开发工具、脚本和 CLI 配置。与业务代码隔离，不参与最终编译。 |
| --- config.yaml | GoFrame CLI 的配置文件，定义编译参数、数据库连接、Docker 构建选项等。 |
| --- hack-cli.mk | Makefile 子模块：负责 `gf` CLI 工具的自动检测与安装。 |
| --- hack.mk | Makefile 子模块：定义项目核心构建命令（build/dao/service/image/deploy 等）。 |
| --- init-project.sh | **模板初始化脚本（macOS/Linux）**：基于当前脚手架创建新项目，自动替换包名、import 路径、应用名，执行 `go mod tidy` 并初始化 Git。 |
| --- init-project.ps1 | **模板初始化脚本（Windows）**：功能与 `.sh` 完全一致，PowerShell 实现。 |
| **internal** | 业务逻辑核心目录。利用 Golang `internal` 特性，此目录下的包仅能被本项目引用，外部模块无法导入，保护核心业务逻辑。 |
| --- cmd | 命令行入口管理。支持启动多个服务（HTTP、WebSocket 等），以及工具命令。 |
| --- consts | 项目常量集中定义。按功能拆分多个文件（app、config、error、http 等），避免魔法值散落各处。 |
| --- controller | 控制器层。接收/解析用户输入参数，调用 Logic/Service 处理业务，返回响应。由 `gf gen ctrl` 根据 `api/` 自动生成骨架。 |
| --- dao | 数据访问对象（Data Access Object）。与底层数据库交互的一层抽象，通常仅包含基础 CURD 方法。目前为空，待 `gf gen dao` 生成。 |
| --- global | 全局变量声明与项目初始化。`init.go` 负责时区设置、链路追踪初始化、缓存适配器配置、系统配置加载等。 |
| --- library | 项目内部扩展库。封装通用的、与业务无关的工具功能，如缓存、上下文管理、响应格式化等。 |
| --- --- cache | 缓存适配器封装。支持 memory/redis/file 三种驱动，通过配置切换。 |
| --- --- contexts | 请求上下文管理。存储请求级别的共享数据（如请求体、模块名），支持后台任务使用分离上下文。 |
| --- --- response | HTTP 响应统一封装。提供 `RJson`、`RXml`、`JsonExit`、`Redirect` 等标准化响应方法。 |
| --- logic | 业务逻辑层。**项目最核心的部分**。按功能子目录组织（middleware、sys、view），通过 `service.RegisterXxx()` 注册到服务接口。 |
| --- --- middleware | 中间件逻辑实现。包含 Ctx 上下文初始化、CORS 跨域、DemoLimit 演示限制、PreFilter 输入过滤、ResponseHandler 响应处理。 |
| --- --- sys | 系统配置逻辑。如系统配置初始化、参数加载等。 |
| --- --- view | 视图渲染逻辑。模板渲染、错误页面展示、面包屑和标题生成。 |
| --- model | 数据模型定义层。管理数据结构对象，与数据表、业务输入输出对应。 |
| --- --- do | DAO 操作对象（Data Object）。用于业务模型与数据库实例模型转换，由 `gf gen dao` 维护，**用户不应手动修改**。 |
| --- --- entity | 数据库实体（Entity）。与数据表一一对应的程序数据结构，由 `gf gen dao` 生成。 |
| --- packed | 静态资源打包目录。`gf build` 时可将 `resource/` 目录打包进可执行文件，实现单二进制部署。 |
| --- router | 路由注册中心。按模块（api/home）分组注册路由，绑定控制器和中间件。 |
| --- service | 服务接口层。定义业务模块的接口契约，由 `gf gen service` 根据 `logic/` 自动生成。**实现依赖倒置，解耦 Controller 与 Logic**。 |
| **logs** | 运行时日志存储目录。按类型分目录（database、logger、server），支持按日期自动轮转。 |
| **manifest** | 编译、部署、运行所需的配置文件清单。与代码分离，便于运维管理。 |
| --- config | 主配置文件 `config.yaml`。包含系统设置、服务器参数、数据库连接、Redis、缓存、日志、国际化等全部配置。 |
| --- deploy/kustomize | Kubernetes 部署配置。使用 Kustomize 管理多环境（base + overlays），支持开发/测试/生产环境差异化部署。 |
| --- docker | Docker 镜像构建相关文件。`Dockerfile` 定义镜像构建步骤，`docker.sh` 提供辅助脚本。 |
| --- i18n | 国际化资源文件目录（预留）。存放多语言翻译文件。 |
| --- protobuf | Protocol Buffers 定义文件（预留）。用于 gRPC 接口定义。 |
| **resource** | 静态资源文件目录。包含 HTML、CSS、JS、图片等，可通过打包注入到二进制中。 |
| --- public | 对外公开访问的静态资源。通过服务器静态文件服务直接提供。 |
| --- template | GoFrame 模板文件。使用 `@{ }` 作为模板定界符，支持动态渲染页面。 |
| **storage** | 本地持久化数据存储。用于文件缓存、开发数据、文档等，通常不纳入版本控制。 |
| --- cache | 文件缓存的存储目录。当 `cache.adapter=file` 时，缓存数据存放于此。 |
| --- dev | 开发环境专用数据。如热重载监控配置等。 |
| --- docs | 项目文档。存放开发规范、目录结构说明等。 |
| **temp** | 编译和运行时的临时文件。存放构建产物、临时日志等，可随时清理。 |
| **utility** | 通用工具包。与 `library` 的区别：`utility` 更偏向纯工具函数（无项目依赖），`library` 依赖项目内部模块。 |
| --- charset | 字符集处理工具。如错误堆栈序列化、特殊字符过滤等。 |
| --- simple | 简易辅助函数集合。如事件系统、系统配置读取、调试模式判断等。 |
| --- validate | 验证工具。提供输入过滤接口、切片包含判断、通用验证函数及单元测试。 |
| **main.go** | 程序唯一入口。初始化全局配置，注册所有逻辑模块，启动命令行服务。 |
| **Makefile** | 项目构建脚本入口。引入 `hack/*.mk` 子模块，提供开发、编译、部署的快捷命令。 |
| **go.mod** | Go Module 依赖管理文件。声明项目模块名和外部依赖版本。 |

### 四、模板初始化命令文档（新增）

本项目支持作为**脚手架模板**复用。通过以下命令可基于当前模板快速创建新项目：

#### 4.1 使用方式

```bash
# macOS / Linux / Windows 通用命令
make init-project name=github.com/company/mynewproject

# 指定输出目录
make init-project name=github.com/company/mynewproject out=./mynewproject
```

#### 4.2 跨平台支持

| 操作系统 | 调用的脚本 | 说明 |
|---------|-----------|------|
| macOS | `hack/init-project.sh` | Bash 脚本，原生支持 |
| Linux | `hack/init-project.sh` | Bash 脚本，原生支持 |
| Windows | `hack/init-project.ps1` | PowerShell 脚本，需安装 Make 或 Git Bash |

#### 4.3 自动替换内容

执行后会自动完成以下操作：

1. **复制模板** — 排除 `.git`、`.idea`、`logs`、`temp`、`go.sum` 等无关文件
2. **替换 `go.mod`** — 将 `module xcross-go-starter` 改为新模块名
3. **替换 import 路径** — 所有 `.go` 文件中的 `"xcross-go-starter/...` 替换为新路径
4. **替换配置项** — `appName`、`title`、`keywords`、`description` 等
5. **替换构建配置** — `hack/config.yaml` 中的可执行文件名和输出路径
6. **替换其他文件** — `README.md`、`Makefile`、`Dockerfile` 中的应用名
7. **执行 `go mod tidy`** — 重新整理 Go 依赖
8. **初始化 Git** — 创建新的 Git 仓库并提交初始版本

#### 4.4 创建后需要手动调整

```bash
cd mynewproject

# 1. 修改数据库连接配置
vim manifest/config/config.yaml

# 2. 修改 GoFrame CLI 的数据库连接（用于 gf gen dao）
vim hack/config.yaml

# 3. 修改 Docker 镜像名和 K8s 部署名
vim Makefile

# 4. 启动服务验证
go run main.go
```

### 五、Makefile 常用命令速查

> 完整命令定义和原理说明参见 `hack/hack-cli.mk` 和 `hack/hack.mk` 中的中文注释。

| 命令 | 作用 | 使用场景 |
|------|------|---------|
| `make build` | 编译项目为二进制可执行文件 | 本地测试编译、CI/CD 构建 |
| `make ctrl` | 根据 `api/` 定义自动生成 Controller | 新增 API 接口后 |
| `make dao` | 根据数据库表生成 DAO/DO/Entity | 数据库表结构变更后 |
| `make service` | 根据 `logic/` 实现生成 Service 接口 | 新增业务方法后 |
| `make enums` | 生成枚举辅助代码 | 新增状态码/类型常量后 |
| `make image` | 构建 Docker 镜像 | 本地构建镜像 |
| `make image.push` | 构建并推送 Docker 镜像到仓库 | CI/CD 流水线 |
| `make deploy` | 部署到 Kubernetes 集群 | 本地联调或生产部署 |
| `make up` | 升级 GoFrame 框架到最新版 | 定期同步安全补丁 |
| `make init-project` | 基于模板创建新项目 | 复用脚手架到新项目 |
| `make cli` | 手动安装/更新 `gf` CLI 工具 | gf 工具缺失或需要更新 |
